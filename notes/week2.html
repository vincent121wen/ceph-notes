第二週：儲存引擎與資料一致性

本週目標：理解 Bluestore 的內部結構、一次寫入請求在 Ceph 內部的完整流程，以及 PG 在故障時如何重新達成一致（peering / recovery / backfill）。

1. Bluestore：Ceph 的儲存引擎

早期 Ceph 使用 Filestore（建立在一般檔案系統如 XFS 之上），後來改為 Bluestore，讓 OSD 直接管理裸磁碟（block device），省去檔案系統層的開銷，效能與資料完整性都更好。Bluestore 主要由三個區塊組成：

區塊用途建議裝置
Data儲存實際物件資料HDD 或 SSD（容量主力）
DB (RocksDB)儲存 metadata：物件位置、屬性、omap 等NVMe/SSD（小但需要低延遲）
WAL
Write-Ahead Log		寫入先落地的日誌，確保斷電後資料一致最快的裝置（通常與 DB 共用同一顆 SSD）

如果 DB/WAL 與 Data 放在同一顆裝置上，稱為「混合部署」；若把 DB/WAL 放在獨立的高速 SSD/NVMe，稱為「分離部署」，可以顯著提升 metadata 密集操作（如小檔案寫入、目錄列舉）的效能。對於你目前的硬體（SAS HDD 為主的 pve-072 等），如果有 SSD 可以考慮把 DB/WAL 分離出來，但這屬於較大的架構調整，第三週會再評估是否值得做。

2. 一次寫入請求的完整流程

理解這個流程，對於排查效能問題與理解一致性保證非常重要。以一個 replica size = 3 的 pool 為例：

  1. Client 計算位置：Client（例如 Proxmox 上的 librbd）根據物件名稱 hash 出 PG，再透過 CRUSH 計算出這個 PG 對應的 OSD 清單，其中第一個是 primary OSD。
  2. 寫入 primary OSD：Client 直接把寫入請求送給 primary OSD（不經過 MON，MON 只負責維護地圖，不參與資料路徑）。
  3. Primary 轉發給 replica：Primary OSD 收到寫入後，會把資料同步轉發給其他 replica OSD。
  4. 等待 ack：所有 replica 都把資料寫入自己的 Bluestore（包含 WAL）後回覆 ack，primary 收到所有 ack 後，才回覆 client「寫入成功」。

這個機制保證了：只要 client 收到寫入成功的回應，資料就已經在所有副本上落地。這也是為什麼 Ceph 在正常情況下可以容忍單一節點故障而不丟資料——但也意味著如果某個副本的寫入延遲較高，會拖慢整個寫入的回應時間。

3. Peering：PG 如何重新「對齊」

當叢集狀態發生變化時（例如 OSD 重啟、加入、移除），對應到這些 OSD 的 PG 需要重新確認「目前最新的資料版本是什麼、各副本是否一致」，這個過程稱為 peering。

Peering 的大致過程：

  • PG 的所有副本（包含 primary）互相交換各自記錄的最新操作日誌（pg log）
  • 選出真正的 primary，並確定哪個副本擁有最新、最完整的資料
  • 找出各副本之間缺失或落後的部分
  • peering 完成後，PG 進入 active 狀態，可以開始處理讀寫請求；缺失的資料則透過 recovery 補齊

Peering 通常很快（數秒內），但如果叢集規模大、PG 數量多，或者同時有大量 OSD 狀態變化（例如整個 host 重啟），peering 的總時間會拉長，期間相關 PG 會短暫無法服務（peering 狀態時 IO 會被阻塞）。

4. Recovery 與 Backfill 的差異

這兩者都是「把缺失的資料補齊」，但適用情境不同：

機制適用情境特點
RecoveryOSD 短暫離線後重新上線（例如重啟），pg log 還在，差異不大只需要補齊「遺漏的那幾筆操作」，速度快、資源消耗較低
BackfillOSD 離線太久導致 pg log 不夠用，或是新加入 OSD、CRUSH 變更導致 PG 需要搬到新位置需要把整個 PG 的資料完整複製過去，類似全量同步，資源消耗較大，可能明顯影響叢集效能

實務上，當你看到 ceph -s 顯示 backfilling 時，通常意味著有大量資料正在搬移（例如 OSD 被移除、新增節點、或 CRUSH map 變更），這時候叢集整體效能會受到影響，Ceph 提供了一些參數（如 osd_max_backfills、osd_recovery_max_active）可以控制 recovery/backfill 的速度與對正常 IO 的影響，第三週會再提到。

5. 實作：在 pve-5 測試環境模擬 OSD 故障

本週的實作建議在 pve-5 的測試 Ceph 叢集上進行（避免影響正式環境）。建議步驟：

  1. 先用 ceph -s 確認測試叢集是 HEALTH_OK
  2. 選定一個 OSD，先用 ceph osd out <id> 把它標記為 out，觀察 ceph -s 與 ceph -w 的即時輸出，記錄狀態變化（rebalance / backfilling 等）
  3. 等待 rebalance 完成後，用 ceph osd in <id> 讓它重新加入，再觀察一次狀態變化
  4. 另外可以嘗試直接停止某個 OSD daemon（systemctl stop ceph-osd@<id>），觀察短時間內叢集如何反應（peering -> degraded -> recovery）

把每一步的 ceph -s 輸出與觀察到的狀態變化記錄到實驗紀錄頁面，特別留意狀態從 active+clean 變成 active+degraded、peering、backfilling，最後回到 active+clean 的完整過程。

6. 本週產出：PG 狀態碼對照表

建議把這週觀察到的狀態碼整理到指令速查表，例如：

  • active+clean — 正常，所有副本一致
  • active+degraded — 至少有一個副本缺失，但 PG 仍可服務讀寫
  • peering — 正在重新確認副本之間的一致性，短暫無法服務
  • recovering — 正在補齊少量缺失資料
  • backfilling — 正在大量搬移資料以達到新的分布
  • undersized — 副本數不足（例如 CRUSH 找不到足夠的位置）
