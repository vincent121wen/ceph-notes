第一週：Ceph 核心架構與健康檢查

本週目標：理解 Ceph 主要元件的角色、資料如何透過 CRUSH 分布、PG 的意義，並對現有叢集做一次完整的健康狀態盤點。

1. Ceph 的主要元件

Ceph 叢集由幾種角色的 daemon 組成，各自負責不同的工作：

元件角色說明
OSD
Object Storage Daemon		實際儲存資料每一個實體/虛擬磁碟通常對應一個 OSD，負責儲存物件、處理讀寫請求、與其他 OSD 互相複製/同步資料、回報自身狀態。
MON
Monitor		叢集狀態仲裁者維護叢集地圖（cluster map：monmap、osdmap、pgmap、crushmap），透過 Paxos 演算法在多個 MON 之間達成一致。MON 數量通常為奇數（3、5），以確保 quorum。
MGR
Manager		監控與管理介面提供 dashboard、Prometheus exporter、容量統計、PG autoscaler 等模組，通常與 MON 同節點部署。
MDS
Metadata Server		CephFS 專用只有使用 CephFS（檔案系統介面）時才需要，負責檔案系統的 metadata（目錄樹、權限等）。若只用 RBD（區塊儲存）則不需要。

對 Proxmox 來說，最常用到的是 RBD（RADOS Block Device），也就是把 Ceph 當作區塊儲存提供給 VM 磁碟。這種情況下 MDS 通常不會啟用。

2. CRUSH 演算法：資料怎麼分佈

CRUSH（Controlled Replication Under Scalable Hashing）是 Ceph 用來決定「一份資料應該放在哪些 OSD 上」的演算法，核心特色是不需要查表，任何 client 都可以透過計算直接得到資料位置，因此沒有中央 metadata 伺服器的瓶頸。

2.1 CRUSH map 的層級結構

CRUSH map 是一個樹狀結構，描述了叢集的實體拓撲，常見層級從下到上大致是：

osd -> host -> chassis -> rack -> row -> datacenter -> root

每一層稱為一個 bucket。Ceph 會根據這個樹狀結構，決定資料的多個副本應該放在不同的「桶子」裡，避免單一節點或機架故障時造成資料完全不可用。

2.2 Failure domain（故障域）

這是 CRUSH 規則裡最重要的概念之一：副本之間至少要分散在哪一層。最常見的設定是 host，意思是同一份資料的多個副本，不會放在同一台主機的不同 OSD 上——這樣即使整台主機掛掉，資料仍然有其他副本可用。

對於你目前 pve-071~075 的環境，可以思考：如果 failure domain 設成 host，那麼複本數（replica size）至少要小於或等於主機數，否則 CRUSH 會找不到足夠的位置放置所有副本，導致 PG 處於 undersized 狀態。

3. PG 與 PGP：為什麼需要它們

如果每個物件都直接對應到 OSD，當叢集擴縮容時，要重新計算「每一個物件」該放在哪裡，計算量會非常龐大。Ceph 因此引入了一個中間層——Placement Group（PG）。

3.1 PG 的角色

  • 每個物件先透過 hash 對應到某一個 PG（物件數量遠大於 PG 數量，所以是多對一）
  • 每個 PG 再透過 CRUSH 對應到一組 OSD（例如 replica size = 3，就對應到 3 個 OSD：1 個 primary + 2 個 replica）
  • 當叢集擴縮容時，只需要重新計算「PG -> OSD」的對應關係，而不是重新計算每一個物件，大幅降低計算與資料搬移的複雜度

3.1.1 用貨櫃比喻理解 PG 與物件的關係

PG 和物件是很容易混淆的兩個概念，用貨運來比喻會比較直覺：

Ceph 概念貨運比喻說明
PG貨櫃負責組織和運送資料的容器，本身不是資料
物件（object）貨櫃裡的貨物實際的資料單位，放在貨櫃（PG）裡
副本（replica）貨櫃的複製份數同一個貨櫃複製 3 份，分放在不同倉庫
OSD倉庫實際存放貨櫃（PG）的地方
host倉庫所在的建築物pve01 / pve02 / pve03
CRUSH貨運排程系統決定貨櫃要送到哪幾個倉庫，並確保同一貨櫃的三份複本不會放在同一棟建築物（failure domain）

以 sblab 為例：33 個貨櫃（PG）、7 件貨物（object）、9 個倉庫（OSD）分布在 3 棟建築物（host）。 大部分貨櫃是空的，只有少數貨櫃裡有貨物（test-vm 的磁碟 image）。 每件貨物都有 3 份複本，分別存放在 pve01、pve02、pve03 各一個倉庫裡。

所以 ceph osd df 裡看到的 4/21 objects degraded (19.048%)，意思是： 21 件貨物副本（7 件 × 3 份）中，有 4 份因為 osd.0（某個倉庫）關閉而暫時遺失， 比例是 4/21 ≈ 19%。這不代表資料遺失，只是暫時少了一份備份。

3.2 PG 數量為何重要

PG 數量太少：每個 PG 承載的資料量過大，OSD 之間負載容易不均，rebalance 時單個 PG 搬移的資料量也會很大。

PG 數量太多：每個 OSD 需要維護的 PG 數量過多，會增加記憶體與 CPU 負擔，peering 時間也會變長。

Ceph 官方建議的經驗值大致是：每個 OSD 承載 100~200 個 PG 左右（這是「該 OSD 上所有 pool 的 PG 數總和，並考慮 replica 數量」後的數字）。現代 Ceph 版本通常會啟用 pg_autoscaler 模組，依據資料量自動調整，但理解背後的邏輯仍然重要，本週先建立概念，第三週會實際檢視。

3.3 PGP 與 PG 的差異（簡述）

PG（pg_num）決定物件被分到多少個群組；PGP（pgp_num）決定這些群組實際對應到 OSD 的方式。通常兩者會設成相同值。當你調大 pg_num 時，Ceph 會先把資料重新分群，但不一定立刻搬移；只有在 pgp_num 也跟著調整後，資料才會根據新的分群實際在 OSD 之間搬移。本週先有這個印象即可，第三週實際操作時會再詳細說明。

4. 實作：盤點 pve-071~075 目前狀態

本週的實作重點是「觀察與記錄」，先不做任何變更。建議依序執行以下指令並記錄輸出（可以記在實驗紀錄頁面）：

# 整體健康狀態
ceph -s

# 每個 OSD 的容量使用情況，觀察是否有明顯不均
ceph osd df

# CRUSH 樹狀結構，確認 host/OSD 對應關係
ceph osd tree

# 目前各 pool 的設定（replica size、PG 數等）
ceph osd pool ls detail

# PG 狀態總覽
ceph pg stat

記錄時可以特別留意：

  • ceph -s 顯示的整體 health 是 HEALTH_OK 還是有 warning？如果有 warning，是什麼類型？
  • ceph osd df 中各 OSD 的使用率（%USE）是否差距很大？最高與最低相差多少？
  • ceph osd tree 中每個 host 下有幾個 OSD，是否平均分布？
  • 目前的 replica size 是多少？failure domain 設定是 host 還是 osd？

5. 本週產出：Ceph 健康檢查 SOP（草稿）

建議在指令速查表頁面開始整理一份「每日/每週該檢查什麼」的清單，例如：

  • 每日：ceph -s 確認 health 狀態、是否有 OSD down
  • 每週：ceph osd df 確認容量分布、是否有 OSD 接近滿載（near full）
  • 異常時：ceph health detail 查看具體警告內容、ceph pg dump 查看異常 PG 的詳細狀態
