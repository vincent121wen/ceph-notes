用貨運比喻理解 PG、物件、OSD 的關係

今天透過破壞性測試（停掉 osd.0）才真正搞懂 PG 和 object 的差別。 之前一直以為 PG 就是資料本身，其實 PG 只是「容器」，資料（object）才是真正放在裡面的東西。

我自己想到的比喻：

Ceph 概念貨運比喻
PG貨櫃（容器，不是貨物本身）
物件（object）貨櫃裡的貨物（實際資料）
副本（replica）貨櫃複製幾份放在不同倉庫
OSD倉庫
host倉庫所在的建築物
CRUSH貨運排程系統，決定貨櫃送到哪些倉庫

19.048% degraded 的計算方式

停掉 osd.0 後看到 4/21 objects degraded (19.048%)， 一開始不知道 21 是怎麼來的，後來理解了：

  • sblab 目前有 7 個 object
  • 每個 object 有 3 份副本
  • 所以副本總數 = 7 × 3 = 21
  • osd.0 掛掉後，有 4 個副本暫時遺失
  • 4 / 21 ≈ 19.048%

這個數字不代表資料遺失，只是少了一份備份。真正危險的門檻是副本數低於 min_size=2，也就是同時有兩個 host 掛掉才會停止服務。

關於 degraded 百分比的門檻

我自己推導：超過 33.3%（約 1/3）通常代表已經有超過 1 個 OSD 出問題， 風險大幅上升。但 Ceph 實際上不是用百分比來判斷，而是追蹤每個 PG 的副本數 是否低於 min_size，百分比只是給人看的參考數字。

PGP 與 CRUSH 的關係

PGP 是 CRUSH 計算系統的輸入參數——CRUSH 不知道你有幾個 PG， 是 PGP 告訴它的。所以 PGP 可以說是「觸發 CRUSH 重新計算的開關」。

  • pg_num：決定有幾個貨櫃（編號規劃）
  • pgp_num：告訴 CRUSH 有幾個貨櫃需要計算對應的 OSD（實際派送執行）
  • 兩者通常設成相同值，代表規劃與執行同步
  • 只調 pg_num 不調 pgp_num → 資料不會真的搬移

破壞性測試觀察到的 PG 狀態

停掉 osd.0 後，PG 出現了兩種新狀態：

  • active+undersized：副本數不足（3→2），但 PG 裡面沒有 object， 所以只顯示 undersized，不顯示 degraded
  • active+undersized+degraded：副本數不足，而且 PG 裡面有 object 受到影響，同時顯示兩個狀態

這讓我理解到：undersized 是 PG 結構的問題，degraded 是 PG 裡面資料的問題， 有資料的 PG 才會同時出現兩個。
